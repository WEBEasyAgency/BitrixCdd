# Версионирование

Ключевая фича — **версионирование конфигов**. Без него при каждом хите структура инфоблоков проверялась бы и обновлялась, что неоправданно дорого.

## Как это работает

1. У каждого конфига инфоблока есть поле `version` (строка, по умолчанию `'1.0'`).
2. При инициализации библиотека проверяет: "я уже обрабатывал конфиг `news` версии `1.0`?"
3. Если да — пропускает. Если нет — синхронизирует и запоминает.
4. Информация хранится в таблице `cdd_iblock_versions` (создаётся автоматически).

## Когда менять версию

Поменял структуру конфига (добавил/удалил свойство, изменил тип) → **увеличь версию**:

```php
'version' => '1.1',  // было '1.0'
```

Это сигнал библиотеке: "конфиг изменился, нужно повторно синхронизировать структуру".

> **Не меняй версию**, если изменил только демо-данные и `strict: false` — демо-данные создаются однократно и не обновляются в этом режиме.

## Матрица поведения need_sync x strict

Эти два флага определяют, насколько агрессивно библиотека синхронизирует БД с конфигом:

| `need_sync` | `strict` | Структура | Демо-данные | Удаление лишнего |
|:-----------:|:--------:|-----------|-------------|------------------|
| `true` | `true` | Всегда (каждый хит) | Принудительно обновляются | Да — свойства и элементы |
| `true` | `false` | По версии | Создаются однократно | Нет |
| `false` | `true` | По версии | Не трогаются | Да — только свойства |
| `false` | `false` | По версии | Не создаются | Нет |

**Рекомендации:**

- **Разработка нового инфоблока**: `need_sync: true, strict: true` — чтобы конфиг был единственным источником правды. Изменил конфиг → следующий хит всё обновит и удалит лишнее.
- **Продакшен**: `need_sync: true, strict: false` — структура создаётся один раз, демо-данные создаются но не затираются, ничего лишнего не удаляется.
- **Когда инфоблок "стабилен"**: `need_sync: false, strict: false` — только создание структуры по версии, никакой синхронизации.

## Что именно делает strict

> **strict — это опасный режим. Читай внимательно.**

При `strict: true` конфиг становится **единственным источником правды**. Всё, чего нет в конфиге — удаляется:

- **Свойства**: кто-то добавил свойство руками в админке — при следующем хите оно удалится.
- **Элементы** (при `need_sync: true`): элементы, которых нет в `demo_data` — удалятся. Если `demo_data` пуст — удалятся **ВСЕ** элементы инфоблока.
- **Типы инфоблоков**: при глобальном `strict: true` инфоблоки управляемых типов, которых нет в конфигах — удалятся.

**Правки в админке не сохранятся.** Если контент-менеджер создал элемент, изменил свойство или добавил новый контент — `strict` перезапишет всё из конфига и удалит лишнее. Админка при `strict: true` — по сути режим "только для просмотра".

> **На продакшене используй `strict: false`.** Режим `strict: true` — только для этапа разработки, когда инфоблок полностью управляется из кода и никто его руками не трогает.

## priority — порядок регистрации

Если один инфоблок ссылается на другой (через `link_iblock_id`), важно чтобы зависимый создался первым:

```php
// articles.php — создаётся первым
'priority' => 100,
'iblock' => ['code' => 'articles', ...],

// news.php — создаётся вторым, может ссылаться на articles
'priority' => 200,
'properties' => [
    'RELATED_ARTICLE' => [
        'type' => 'E',
        'link_iblock_id' => 'articles',  // code, не ID
    ],
],
```

По умолчанию `priority = 500`. Меньше = раньше.
