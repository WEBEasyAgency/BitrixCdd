# Версионирование

Ключевая фича -- **версионирование конфигов**. Без него при каждом хите структура инфоблоков проверялась бы и обновлялась, что неоправданно дорого.

## Как это работает

1. У каждого конфига инфоблока есть поле `version` (строка или число, по умолчанию `'1.0'`).
2. При инициализации библиотека проверяет: "я уже обрабатывал конфиг `news` версии `1.0`?"
3. Если да -- пропускает. Если нет -- синхронизирует и запоминает.
4. Информация хранится в таблице `cdd_iblock_versions` (создается автоматически).

## Когда менять версию

Поменял структуру конфига (добавил/удалил свойство, изменил тип) -- **увеличь версию**:

```php
'version' => '1.1',  // было '1.0'
```

Это сигнал библиотеке: "конфиг изменился, нужно повторно синхронизировать".

## sync_mode и версионирование

Режим синхронизации (см. [configuration.md](configuration.md)) определяет, что происходит при новой версии:

| sync_mode | При новой version | При повторном хите той же version |
|-----------|-------------------|----------------------------------|
| `off` | Структура | Ничего |
| `soft` | Структура + демо (create only) | Ничего |
| `ensure` | Структура + демо (create/дозаполнить) | Ничего |
| `once` | Структура + демо (force) + удаление лишнего | Ничего |
| `danger` | Все | Все (version игнорируется) |

В режимах `off`, `soft`, `ensure`, `once` версионирование работает как gate: обработка происходит один раз на version, потом пропускается. В `danger` версионирование отключено -- синхронизация при каждом хите.

### Особенности по режимам

**`soft`**: при бампе version обновляется структура. Демо-данные создаются только если элемент с таким кодом не существует. Если элемент уже был создан на прошлой версии -- он не трогается.

**`ensure`**: при бампе version обновляется структура. Для демо-данных: если элемента нет -- создается. Если есть, но часть свойств пустая -- дозаполняются из конфига. Уже заполненные свойства не перезаписываются.

**`once`**: при бампе version -- полная принудительная синхронизация. Структура обновляется, лишние свойства удаляются. Если в конфиге есть `demo_data` -- данные перезаписываются, лишние элементы удаляются. Если `demo_data` пуст -- элементы не трогаются. После применения -- стоп до следующего бампа.

## Legacy: need_sync и strict

Старые свойства `need_sync` и `strict` поддерживаются для обратной совместимости. Если `sync_mode` не указан, он вычисляется автоматически:

| need_sync | strict | sync_mode |
|:---------:|:------:|-----------|
| true | true | `danger` |
| true | false | `soft` |
| false | true | `once` |
| false | false | `off` |

Если указан `sync_mode` -- он имеет приоритет над `need_sync`/`strict`.

## priority -- порядок регистрации

Если один инфоблок ссылается на другой (через `link_iblock_id`), важно чтобы зависимый создался первым:

```php
// articles.php
'priority' => 100,
'iblock' => ['code' => 'articles', ...],

// news.php
'priority' => 200,
'properties' => [
    'RELATED_ARTICLE' => [
        'type' => 'E',
        'link_iblock_id' => 'articles',  // code, не ID
    ],
],
```

По умолчанию `priority = 500`. Меньше = раньше.
