# Конфигурация

## Структура конфигов

```
local/config/
├── global.php              # Глобальные настройки (мержатся во все конфиги)
├── iblock_types/           # Типы инфоблоков (один файл = один тип)
│   └── content.php
└── iblocks/                # Инфоблоки (один файл = один инфоблок)
    └── news.php
```

Путь к конфигам можно переопределить:

```php
$app->initialize([
    'config_dir' => $_SERVER['DOCUMENT_ROOT'] . '/custom/config',
]);
```

## sync_mode

Управляет синхронизацией структуры и данных. По умолчанию `'soft'`.

| Режим | Структура | Демо-данные | Удаление лишнего |
|----------|-----------|-------------|------------------|
| `off` | Раз на version | Никогда | Нет |
| `soft` | Раз на version | Раз (create only) | Нет |
| `ensure` | Раз на version | Создать если нет, дозаполнить пустое | Нет |
| `once` | Раз на version | Раз на version (force) | Да |
| `danger` | Каждый хит | Каждый хит (force) | Да |

- **`off`** -- создает инфоблок и свойства при первом запуске (или при бампе version). Демо-данные игнорируются. Для инфоблоков, данные которых управляются только через админку.

- **`soft`** -- как `off`, но дополнительно создает элементы из `demo_data` один раз. Если элемент с таким кодом уже существует, он не трогается. При бампе version обновляется только структура (новые свойства), данные остаются как есть.

- **`ensure`** -- как `soft`, но умнее: если элемент существует, но часть свойств пустая -- дозаполняет их из `demo_data`. Уже заполненные свойства не перезаписываются. Подходит для деплоя на сервер, где инфоблок может быть в частичном состоянии.

- **`once`** -- полная принудительная синхронизация при смене version: обновляет структуру, перезаписывает данные, удаляет свойства и элементы, которых нет в конфиге. После применения -- стоп до следующего бампа version. Для случаев когда нужно привести инфоблок к конкретному состоянию.

- **`danger`** -- как `once`, но без версионирования: синхронизация происходит при каждом хите. Данные из `demo_data` перезаписывают админские правки. Только для разработки.

### Выбор режима

```
Прод, данные в админке           -> off
Прод, первичная заливка          -> soft
Прод, гарантия доставки данных   -> ensure
Миграция к конкретному состоянию -> once (бампнуть version)
Разработка                       -> danger
```

### Legacy-совместимость

Старые свойства `need_sync` и `strict` поддерживаются:

| need_sync | strict | sync_mode |
|-----------|--------|-----------|
| false | false | `off` |
| true | false | `soft` |
| false | true | `off` |
| true | true | `danger` |

Если указан `sync_mode` -- он имеет приоритет над `need_sync`/`strict`.

## Тип инфоблока

Каждый инфоблок привязан к типу. Тип задаётся в `global.php` через `default_iblock_type` или в конфиге инфоблока через `iblock.type`.

Если тип не существует в БД, он создаётся автоматически с русскими лейблами и `sections: 'Y'`. Для типа `content` название будет "Контент".

Для тонкой настройки типа создай файл `local/config/iblock_types/{id}.php`:

```php
<?php
return [
    'id'       => 'content',
    'sections' => 'Y',
    'lang'     => [
        'ru' => [
            'NAME'         => 'Контент',
            'SECTION_NAME' => 'Разделы',
            'ELEMENT_NAME' => 'Элементы',
        ],
    ],
];
```

## Конфиг инфоблока

```php
<?php
return [
    'version'   => '1.0',       // По умолчанию '1.0' (подробнее в versioning.md)
    'sync_mode' => 'ensure',    // По умолчанию 'soft'
    'priority'  => 100,         // По умолчанию 500 (меньше = раньше)

    'iblock' => [
        'code' => 'news',       // Обязательно
        'name' => 'Новости',    // Обязательно
        'type' => 'content',    // По умолчанию из default_iblock_type в global.php
        'sort' => 100,          // По умолчанию 500
    ],

    'iblock_fields' => [        // Мержится с global.php
        'CODE' => ['IS_REQUIRED' => 'Y'],
    ],

    'include_standard_fields' => ['NAME', 'PREVIEW_PICTURE'],

    'properties' => [
        'AUTHOR' => [
            'name'     => 'Автор',      // Обязательно
            'type'     => 'S',          // По умолчанию 'S' (строка)
            'sort'     => 100,
            'required' => true,         // По умолчанию false
            'multiple' => false,        // По умолчанию false
        ],
    ],

    'demo_data' => [...],   // Ключ demoData тоже поддерживается
];
```

## Свойства (properties)

Обязательное поле -- только `name`. Всё остальное имеет умолчания.

### Типы свойств

| `type` | Описание | Примечания |
|--------|----------|------------|
| `S` | Строка (по умолчанию) | С `user_type` становится Date, DateTime, HTML |
| `N` | Число | Целое или дробное |
| `L` | Список (enum) | Требует ключ `values` |
| `F` | Файл | Можно ограничить расширения через `file_type` |
| `E` | Привязка к элементу | Можно указать `link_iblock_id` (код инфоблока) |
| `G` | Привязка к разделу | Можно указать `link_iblock_id` (код инфоблока) |

### Свойства с user_type

```php
'DATE_START'   => ['name' => 'Дата начала',     'user_type' => 'Date'],
'PUBLISHED_AT' => ['name' => 'Дата публикации', 'user_type' => 'DateTime'],
'BODY'         => ['name' => 'Текст',           'user_type' => 'HTML'],
```

### Свойство типа "Список" (L)

Два формата значений, оба ключа (`values` и `VALUES`) поддерживаются:

```php
// Плоский список строк (SORT нумеруется автоматически: 10, 20, 30...)
'STATUS' => [
    'name'   => 'Статус',
    'type'   => 'L',
    'values' => ['Черновик', 'Опубликовано', 'Архив'],
],

// Полный формат
'STATUS' => [
    'name'   => 'Статус',
    'type'   => 'L',
    'VALUES' => [
        ['VALUE' => 'Черновик',     'SORT' => 10, 'DEF' => 'Y'],
        ['VALUE' => 'Опубликовано', 'SORT' => 20],
        ['VALUE' => 'Архив',        'SORT' => 30],
    ],
],
```

## include_standard_fields

Управляет тем, какие стандартные поля Битрикса попадают в выдачу `ElementDataExtractor`.

| Значение | Результат |
|----------|-----------|
| `false` (по умолчанию) | Только свойства из `properties` |
| `true` | Полный набор стандартных полей (см. список ниже) |
| массив | Только перечисленные стандартные поля |

Стандартные поля при `true`:

| Поле | Описание | Трансформация |
|------|----------|---------------|
| `NAME` | Название элемента | как есть |
| `SORT` | Сортировка | как есть |
| `PREVIEW_TEXT` | Текст анонса | как есть |
| `PREVIEW_PICTURE` | Картинка анонса | ID -> URL |
| `DETAIL_TEXT` | Детальный текст | как есть |
| `DETAIL_PICTURE` | Детальная картинка | ID -> URL |

```php
'include_standard_fields' => true,                        // все 6 полей
'include_standard_fields' => ['NAME', 'PREVIEW_PICTURE'], // только указанные
```

## Расширенная настройка полей (fields)

Для случаев, когда автоматический вывод типов не подходит (например, нужен `file_info` вместо `file`), можно задать секцию `fields` явно. В этом случае `include_standard_fields` игнорируется.

```php
'fields' => [
    'NAME'     => ['type' => 'standard'],
    'AUTHOR'   => ['type' => 'property', 'property_type' => 'string'],
    'DOCUMENT' => ['type' => 'property', 'property_type' => 'file_info'],
],
```

Автоматический вывод `property_type`:

| property type | user_type | property_type |
|---------------|-----------|---------------|
| S | - | string |
| S | HTML | text |
| N | - | string |
| L | - | enum |
| F | - | file |
| E | - | element |
| G | - | element |

Если `property_type` не указан в `fields`, он выводится из `properties` по этой таблице.

## Типограф (typograph)

Автоматическая расстановка неразрывных пробелов после предлогов и союзов. Предотвращает перенос предлога на предыдущую строку ("в", "на", "и", "с", "за", "из", "до", "по", "от", "для", "при", "без", "под", "над", "про", "через" и др.).

Включается для отдельных свойств через `'typograph' => true`:

```php
'properties' => [
    'TITLE' => ['name' => 'Заголовок', 'typograph' => true],
    'BODY'  => ['name' => 'Текст', 'user_type' => 'HTML', 'typograph' => true],
    'CODE_SNIPPET' => ['name' => 'Код'],  // без typograph
],
```

Результат:
- `Косметика для ногтей и кожи` -> `Косметика для&nbsp;ногтей и&nbsp;кожи`
- HTML-теги не затрагиваются, обрабатывается только текст между ними

Для стандартных полей typograph задаётся в явной секции `fields`:

```php
'fields' => [
    'NAME' => ['type' => 'standard', 'typograph' => true],
],
```

Обработка происходит при чтении через `ElementDataExtractor`. Данные в БД не изменяются.

Типограф также доступен как отдельный сервис:

```php
$typographer = new \BitrixCdd\Services\Typographer();
$text = $typographer->process('Мы работаем в Москве и в Питере');
// "Мы работаем в&nbsp;Москве и&nbsp;в&nbsp;Питере"
```

## Программная регистрация

```php
$configService = $app->getService('iblock.config');

$configService->registerIblockFromConfig([
    'iblock' => [
        'code' => 'dynamic_catalog',
        'name' => 'Динамический каталог',
    ],
    'properties' => [
        'PRICE' => ['name' => 'Цена', 'type' => 'N'],
    ],
]);
```

Если тип инфоблока не существует, он будет создан автоматически.
