# DevOps Audit

Аудит текущего репозитория на соответствие devops-требованиям.

## Определение контекста

Перед запуском проверок определи тип проекта:

1. **Bitrix** - есть `local/` директория, `bitrix/` в .gitignore, или файлы с `Bitrix\` namespace
2. **WordPress** - есть `wp-content/`, `wp-config.php`
3. **Laravel** - есть `artisan`, `app/Http/`
4. **Generic** - всё остальное

Bitrix-специфичные проверки (категория 5) запускай только для Bitrix-проектов.

Если пользователь передал URL сервера или он известен из проекта (.gitlab-ci.yml, CLAUDE.md) - запусти также проверки доступности sensitive-файлов (категория 6).

## Выполнение проверок

Запускай проверки максимально параллельно. Каждая проверка описана ниже с инструментом и критичностью.

### Категория 1: Git-конфигурация

**1.1. .gitignore существует**
- Glob: `.gitignore` в корне проекта
- Критичность: ВЫСОКАЯ

**1.2. .gitignore - обязательные исключения**
Прочитай .gitignore и проверь наличие (прямо или через паттерны):
- `.env` - КРИТИЧНО если отсутствует
- `node_modules/` - ВЫСОКАЯ
- `vendor/` - ВЫСОКАЯ (если проект использует composer)
- Артефакты сборки: `*.min.css`, `*.min.js`, `dist/` - ВЫСОКАЯ
- IDE: `.idea/`, `.vscode/` - НИЗКАЯ
- OS: `.DS_Store`, `Thumbs.db` - НИЗКАЯ

Учитывай что исключения могут быть заданы паттернами (например `*.min.*` покрывает и css и js).

**1.3. develop ветка существует**
- Bash: `git branch -a | grep -E '(^|\s)develop$|/develop$'`
- Критичность: ВЫСОКАЯ

**1.4. Артефакты сборки не закоммичены**
- Bash: `git ls-files '*.min.css' '*.min.js'` и `git ls-files 'dist/'`
- Наличие результатов = проблема
- Критичность: СРЕДНЯЯ

**1.5. .env не отслеживается git**
- Bash: `git ls-files .env`
- Если файл tracked - КРИТИЧНО, пароли в истории git
- Критичность: КРИТИЧЕСКАЯ

### Категория 2: Документация

**2.1. README.md существует**
- Glob: `README.md` в корне
- Критичность: ВЫСОКАЯ

**2.2. README.md содержит инструкцию по сборке**
- Прочитай README.md, проверь наличие команд сборки (npm, composer, gulp, webpack, docker, git pull, или аналоги)
- Критичность: СРЕДНЯЯ

**2.3. env.example существует**
- Glob: `.env.example` или `env.example`
- Проверяй только если проект использует .env (есть .env в .gitignore)
- Критичность: СРЕДНЯЯ

### Категория 3: Безопасность (.htaccess)

Актуально для Apache-проектов. Если проект использует nginx (нет .htaccess, есть nginx.conf) - укажи что проверки .htaccess неприменимы и нужна ручная проверка nginx-конфига.

**3.1. .htaccess существует**
- Glob: `.htaccess` в корне
- Критичность: ВЫСОКАЯ

**3.2. Dotfiles заблокированы**
- Grep в .htaccess: паттерн блокировки файлов начинающихся с точки (например `FilesMatch "^\."` или аналог)
- Без этого правила `.env` доступен по URL
- Критичность: КРИТИЧЕСКАЯ

**3.3. Служебные файлы заблокированы**
- Grep в .htaccess: блокировка `package.json`, `composer.json`, `*.md` файлов
- Критичность: ВЫСОКАЯ

**3.4. Служебные директории заблокированы**
- Grep в .htaccess: блокировка `.git`, `node_modules`, `.gitlab`
- Без этого `.git/config` и вся история доступны извне
- Критичность: КРИТИЧЕСКАЯ

### Категория 4: CI/CD

**4.1. .gitlab-ci.yml существует**
- Glob: `.gitlab-ci.yml`
- Если нет - проверь наличие других CI-конфигов (.github/workflows/, Jenkinsfile, .circleci/)
- Критичность: ВЫСОКАЯ

### Категория 5: Bitrix-специфичные (только для Bitrix-проектов)

**5.1. Хардкод путей к шаблону**
- Grep в `local/templates/` и `local/components/`: паттерн `addCss.*"/local/templates/` или `addJs.*"/local/templates/`
- Должно быть: `SITE_TEMPLATE_PATH`
- Критичность: СРЕДНЯЯ

**5.2. Ассеты вне /local/assets/**
- Grep в PHP-файлах: `addCss.*"/(assets|css)/` или `addJs.*"/(assets|js)/` (без /local/ префикса)
- Правильно: `/local/assets/`
- Критичность: НИЗКАЯ

**5.3. Голые теги вместо Asset::getInstance()**
- Grep в `local/templates/` PHP-файлах: `<link rel=.stylesheet` или `<script src=`
- Не считай за ошибку если это в HTML-вёрстке (app/pages/, layout/) - только в Bitrix-шаблонах
- Критичность: СРЕДНЯЯ

**5.4. Хардкод ID инфоблоков**
- Grep в `local/components/` и `local/templates/`: паттерны `\$iblockId\s*=\s*\d+`, `IBLOCK_ID.*=>\s*\d+`, `'IBLOCK_ID'\s*=>\s*\d+`
- Исключи файлы конфигурации (config/iblocks/) и комментарии
- Критичность: СРЕДНЯЯ

**5.5. Растровый логотип**
- Grep в `local/templates/`: `logo\.(png|jpg|jpeg|gif|webp)`
- Рекомендация: SVG
- Критичность: НИЗКАЯ

### Категория 6: Проверка сервера (опционально, если есть URL)

Если известен URL сервера, проверь доступность sensitive-файлов через curl. Если есть basic auth - используй его.

```bash
# Каждый URL должен вернуть 403 или 404, НЕ 200
curl -s -o /dev/null -w "%{http_code}" https://<domain>/.env
curl -s -o /dev/null -w "%{http_code}" https://<domain>/.git/config
curl -s -o /dev/null -w "%{http_code}" https://<domain>/.gitlab-ci.yml
curl -s -o /dev/null -w "%{http_code}" https://<domain>/package.json
curl -s -o /dev/null -w "%{http_code}" https://<domain>/README.md
```

HTTP 200 на любом из этих URL = КРИТИЧЕСКАЯ проблема безопасности.

## Формат отчёта

Выведи результат в таком формате (без эмодзи):

```
## Аудит репозитория: [имя проекта]
Тип проекта: [Bitrix/WordPress/Laravel/Generic]
Дата: [дата]

### 1. Git-конфигурация
- [x] .gitignore существует
- [x] .env исключён из git
- [ ] ПРОБЛЕМА: node_modules/ не в .gitignore
  > Добавьте `node_modules/` в .gitignore
- [x] develop ветка существует
...

### 2. Документация
...

### 3. Безопасность
...

### 4. CI/CD
...

### 5. Bitrix   [если применимо]
...

### 6. Сервер   [если проверялся]
...

---

### Итого
Пройдено: N из M
Критичных проблем: X
Проблем средней важности: Y
Рекомендаций: Z
```

Правила оформления:
- `[x]` - проверка пройдена
- `[ ]` с пометкой ПРОБЛЕМА/КРИТИЧНО - не пройдена, с кратким описанием и рекомендацией по исправлению
- `[-]` - проверка неприменима (например, Bitrix-проверки для не-Bitrix проекта)
- Группируй по категориям
- В итоге указывай точные числа
- Критичные проблемы выделяй жирным
